# 🆘 Система техподдержки (тикеты через форум-топики)

Назад к [[00_Index]]. Модули: `handlers/support.py`, `database/models.py`
(`SupportTicket`), `database/repository.py`, `config.SUPPORT_CHAT_ID`. Открыта всем
(не под SubscriptionMiddleware).

Двусторонний мост «личка ↔ супергруппа поддержки»: каждому пользователю — свой
**топик (тема)** в супергруппе. Сообщения юзера копируются в его топик; ответы
админов из топика — обратно юзеру в личку.

---

## Требования к супергруппе

| Что | Зачем |
|-----|-------|
| `SUPPORT_CHAT_ID` (env, **`-100…`**) | id супергруппы поддержки; `0` → поддержка выключена (хендлеры мягко сообщают) |

> ⚠️ **Парсинг id (важно!).** ID супергруппы **отрицательный** (`-100…`). Обычный
> `_env_int` через `.isdigit()` отбрасывал минус → значение падало в фолбэк `0` и
> поддержка не работала. Поэтому `config._parse_support_chat_id()` парсит `int(...)`
> со знаком и **предупреждает в логах**, если id задан без `-100` (с положительным
> id бот не сможет писать в группу).
| Включены **топики (Forum)** | бот создаёт по теме на юзера (`create_forum_topic`) |
| Бот — **админ** с правом «Manage Topics» | иначе `create_forum_topic` падает (ловим → «Попробуйте позже») |
| **Privacy Mode = OFF** (@BotFather) | иначе бот НЕ видит ответы админов в группе → мост в одну сторону |

---

## Модель `support_tickets` (одна строка = один юзер)

| Поле | Тип | Назначение |
|------|-----|-----------|
| `telegram_id` | BigInt, **PK** | пользователь (прямой поиск юзер→топик) |
| `topic_id` | Integer, **index** | `message_thread_id` темы в группе (обратный поиск топик→юзер) |
| `created_at` | DateTime(tz), now | дата открытия тикета |

Репозиторий: `get_support_ticket(tg)` (прямой), `get_ticket_user_by_topic(topic_id)`
(обратный), `create_support_ticket(tg, topic_id)` (идемпотентно — обновляет topic_id,
не плодит строки).

## Флоу

### Пользователь → поддержка (`handlers/support.py`) — режим ЧАТ-КОМНАТЫ
```
[🆘 Техподдержка] (reply-кнопка меню) → on_support_button:
  FSM → SupportState.waiting_for_message
  показывает reply-клавиатуру с ОДНОЙ кнопкой «❌ Выйти из чата поддержки» (_support_kb)
  «Напишите ваш вопрос… Можно несколько сообщений подряд. Когда закончите — Выйти.»

[«❌ Выйти из чата поддержки»] → on_exit_support (зарегистрирован ПЕРЕД основным):
  state.clear(); «Вы вышли из чата с поддержкой…»; main_menu_kb()

[любое другое сообщение в состоянии] → on_user_support_message:
  if text == «❌ Выйти…»: → on_exit_support (подстраховка, текст НЕ уйдёт в топик)
  topic_id = get_support_ticket(tg)?.topic_id
  если None: topic = bot.create_forum_topic(SUPPORT_CHAT_ID,
                       name="🎫 {first_name} (ID: {user.id})")   # компактно, ≤128 симв.
             create_support_ticket(...)
             # + в топик: карточка-анкета (имя/ID/@username/дата) → pin_chat_message
             #   (детали вынесены из названия темы в закреплённое сообщение)
  message.copy_to(SUPPORT_CHAT_ID, message_thread_id=topic_id)   # текст/фото/документ
  «✅ Отправлено в поддержку…»  — СОСТОЯНИЕ НЕ СБРАСЫВАЕТСЯ (диалог продолжается)
```

> 🔁 **Непрерывный диалог.** Состояние `waiting_for_message` удерживается, пока юзер
> сам не нажмёт «❌ Выйти». Каждое сообщение в этом режиме уходит в его топик — не
> нужно жать кнопку поддержки заново. На фатальных сбоях (нет `SUPPORT_CHAT_ID`,
> упал `create_forum_topic`) состояние сбрасывается, чтобы не запереть юзера.

### Поддержка → пользователь (`on_admin_reply`)
```
@router.message(F.chat.id == SUPPORT_CHAT_ID, F.message_thread_id.is_not(None),
                ~F.forum_topic_created)          # сервисные сообщения о создании темы — мимо
  если from_user.is_bot: return                  # копии бота не пересылаем (и в апдейтах их нет)
  user_id = get_ticket_user_by_topic(message.message_thread_id)
  если user_id: message.copy_to(chat_id=user_id) # ответ (текст/медиа) в личку юзеру
```

- **`copy_to`** (Bot API copyMessage) переносит любой тип контента без шапки
  «forwarded from» — и в топик, и обратно юзеру.
- Свои сообщения бот в апдейтах не получает, поэтому копии юзерских сообщений в
  топике не зацикливаются; человеческие ответы админов — триггерят `on_admin_reply`.
- Все сетевые вызовы в try/except: бот не падает, если юзер заблокировал бота или
  группа недоступна. Доступ к БД — через `to_thread` (sync-сессии).

## Краевые случаи

- `SUPPORT_CHAT_ID == 0` → «Техподдержка временно недоступна» (без создания топика).
- `create_forum_topic` упал (бот не админ / топики выключены) → «Не удалось создать
  обращение. Попробуйте позже», тикет не пишется.
- Повторное обращение того же юзера → топик переиспользуется (создаётся ровно один раз).
- Сообщение в топике без тикета (ручная тема) → `user_id is None` → игнор.
