# 📊 Анализ конкурентов из карточки товара

Назад к [[00_Index]]. Premium-фича. Модули: `handlers/competitors_analytics.py`,
`services/analytics_scoring.py` (Индекс Качества). `services/uzum_public_api.py`
(внешний парсер) — **временно НЕ используется** (блокировки Uzum).

Кнопка **`📊 Анализ конкурентов`** в фото-карточке SKU ([[06_Bot_Flow_&_States]],
`product_card_kb`, callback `prod:analyze_competitors:<sku_id>`) → бот подбирает
похожие товары, считает их Индекс Качества и выдаёт сравнительный отчёт.

> 🔀 **ГИБРИДНЫЙ ДВИЖОК (актуально).** Из-за блокировок внешний парсинг Uzum
> отключён. Поиск конкурентов переведён на **нашу локальную БД** (товары, загруженные
> ДРУГИМИ пользователями бота) + **умный Mock-генератор** реалистичных «топ-продавцов»
> при нехватке локальных данных. Сеть в этом флоу не используется — отчёт строится
> стабильно в любых условиях.

## Флоу (хэндлер `on_analyze_competitors`, Premium-only)

```
[📊 Анализ конкурентов] prod:analyze_competitors:<sku_id>
1. _load_seller (to_thread, БД): telegram_id + title + product_id + local_metrics
2. status «🔎 Подбираю похожие товары…» → _build_competitor_report (БД, без сети):
   a. search_local_competitors(tg, title, sku_id, 3)   → похожие из user_products
      (ДРУГИХ юзеров, ILIKE по словам названия, без своего sku_id, дедуп по sku_root);
      каждой строке присваиваются реалистичные метрики «топа» (в БД их нет).
   b. если найдено < 3 → generate_mock_competitors(title, добивка) — синтетические
      карточки (рейтинг 4.6–4.9, 4–8 фото, описание 300–800, 6–12 характеристик).
   c. get_product_quality_score для КАЖДОГО конкурента.
   d. seller_norm = local_metrics селлера (фото=1 если есть превью, рейтинг/опис/хар-ки=0).
   e. render_report(...) → сравнительный отчёт со средними и рекомендациями.
3. status.edit_text(report)
```

- **`search_local_competitors`** (async) — обёртка над sync-SQL
  `repository.find_local_competitors` через `to_thread` (в проекте доступ к БД
  синхронный, без async-пула asyncpg; параметр `db_pool` из ТЗ заменён на
  `session_scope`).
- **Mock — «умный»:** названия строятся от названия товара селлера
  («{Название} Pro/Premium/Lux/…»), метрики — правдоподобные показатели сильной
  карточки. Mock и локальные конкуренты дают одинаковый нормализованный формат.

Роутер `competitors` гейтится `SubscriptionMiddleware` (в `register_handlers`),
поэтому кнопка реально работает только у Premium/админов (free → оффер Premium,
см. [[10_Billing_&_Subscriptions]]).

## Индекс Качества — `get_product_quality_score(product_data)`

Чистая функция (0–100), `services/analytics_scoring.py`. Вход — нормализованные
метрики `{rating, photos_count, description_len, characteristics_count}`:

| Параметр | Баллы |
|----------|-------|
| **Рейтинг** (40) | ≥4.8 → 40 · 4.5–4.7 → 25 · <4.5 (в т.ч. нет) → 10 |
| **Фото** (20) | ≥5 → 20 · 2–4 → 10 · 1 → 2 · 0 → 0 |
| **Описание** (20) | >300 симв → 20 · 100–300 → 10 · <100 → 0 |
| **Характеристики** (20) | >5 → 20 · иначе 5 |

`normalize_uzum_card(raw)` защитно вытаскивает метрики из «сырой» карточки
(пробует разные имена полей: `photos/images/gallery`, `characteristics/attributes`,
`description/...`); чего нет — считает нулём, исключений не бросает.

## Публичный парсер — `services/uzum_public_api.py` (лёгкий web-парсинг)

Отказались от тяжёлого Playwright/Chromium в пользу httpx + bs4: быстро, легко,
масштабируемо. Мобильный app-API (`api.uzum.uz`/`graphql.uzum.uz`) требует app-токен
`x-iid`/подпись (401), поэтому **читаем публичные WEB-страницы** `https://uzum.uz/ru`
без браузера и достаём данные из HTML.

- `search_competitors_on_uzum(query, limit=3) → list[int]` —
  `GET {UZUM_WEB_BASE}/search?query=…`; productId из JSON-стейта, иначе из ссылок
  `/product/<id>` (`re.finditer`).
- `fetch_product_card(product_id) → dict | None` — **мобильный API**
  `GET {UZUM_PUBLIC_API_BASE}/product/{id}` → **чистый JSON** через `_get_json`,
  данные тянутся безопасно через `.get()` (title/rating/`photos[]`/`attributes`).
  Web-страница `uzum.uz/ru/product/{id}` НЕ используется — её WAF гонял в цикл
  редиректов («Exceeded maximum allowed redirects»); мобильный JSON-эндпоинт
  редиректов не делает. На 401/403 (нужна подпись) или не-JSON → None → хэндлер
  уходит на локальные метрики БД («Экспресс-аудит»), без сетевой ошибки. (bs4 для
  карточки больше не нужен — он остался только в истории web-парсинга.)
- `fetch_product_cards(ids)` — `asyncio.gather`, конкуренция ограничена семафором.

### 🚦 Защита сервера: `CONCURRENCY_LIMITER = asyncio.Semaphore(5)`

Модульный семафор оборачивает **каждый** сетевой запрос (`_get_html`): даже если
30+ юзеров одновременно нажмут «📊 Анализ конкурентов», к Uzum уйдёт не более **5**
запросов одновременно — стабильность сервера и меньше подозрений у анти-бота.
Проверено: при 20 параллельных запросах пиковая конкурентность = ровно 5.

### Клиент `_make_client()` — HTTP/2 + прокси + мобильные заголовки

```python
proxy = PROXY_URL or None        # http://user:pass@ip:port (config); пусто → без прокси
httpx.AsyncClient(http2=True, proxy=proxy, follow_redirects=True,
                  headers=MOBILE_HEADERS, timeout=5.0)
```
- **Ротируемый прокси** — `config.PROXY_URL` (env). ⚠️ httpx **0.28+**: параметр
  `proxies=` удалён, используем **`proxy=`** (одиночный, на все схемы).
- **HTTP/2** (пакет `h2`; нет — fallback на 1.1) + `MOBILE_HEADERS` (мимикрия под
  Android-приложение). Таймаут 5 с.
- **Дебаг-лог:** статус ≠ 200 → `log.warning(url, status, text[:200])`.

### Извлечение данных из HTML

`_parse_product_html(html)`: сперва **JSON-стейт инициализации** — `_extract_state_json`
ищет `window.__INITIAL_STATE__` / `__NUXT__` / `__NEXT_DATA__` / `__PRELOADED_STATE__`
и парсит объект **балансировкой скобок** (`_json_after`, корректно по вложенности и
строкам с `{`/`}`). В нём «всё в чистом JSON» — рейтинг, фото, характеристики
(`_find_product_node` находит узел товара). Если стейта нет — **bs4-фолбэк**:
`h1`/`og:title` → название, `meta[name=description]` → описание, `img` с
`images.uzum.uz` → фото. Результат скорит `normalize_uzum_card` + `get_product_quality_score`.

> ⚠️ **РЕАЛЬНОСТЬ.** `uzum.uz/ru/product/{id}` под анти-ботом уходит в **цикл
> редиректов** (на captcha/логин) — httpx ловит «Exceeded maximum allowed redirects»,
> парсер возвращает None (бот не падает, фича → «Экспресс-аудит»). Для боевой работы
> нужен **рабочий прокси** (`PROXY_URL`) с резидентными/мобильными IP и/или валидная
> сессия. Архитектура (Semaphore, HTTP/2, прокси, bs4) готова — подставляется прокси
> через env без изменения кода. Скоринг/рендер от сети не зависят, протестированы.

## Отчёт — `render_report(...)` (чистая функция)

**Когда конкуренты найдены:**

```
📊 Анализ конкурентов: <название>
───────────────────
🏆 Топ-N конкурентов на Uzum Market
├── 📷 фото в среднем: 8
├── ⭐ рейтинг в среднем: 4.8
└── 🏅 Индекс Качества: 95/100

📦 Ваша карточка
├── 🏅 Индекс Качества: 17/100
├── 📷 фото: 1
└── ⭐ рейтинг: 4.3

💡 Вывод: у топ-конкурентов в среднем 8 фото и рейтинг 4.8.
Ваша карточка набрала 17/100 баллов.
Рекомендации: 📷 добавьте фото · 📝 расширьте описание · 📋 добавьте характеристики · ⭐ рейтинг
```

Рекомендации (`_recommendations`) генерятся по слабым местам относительно
средних показателей конкурентов (мало фото / короткое описание / мало
характеристик / низкий рейтинг); если карточка не уступает — «✅ так держать!».

**Деградация / «Экспресс-аудит карточки» (конкурентов нет):** если `comp_norms`
или `comp_scores` пусты (парсинг упал/0 результатов), отчёт **полностью**
переключается в режим одиночного аудита:
- заголовок «🔍 Экспресс-аудит карточки: <название>» (не «📊 Анализ конкурентов»);
- свой Индекс Качества (по локальным данным: превью-фото; рейтинг/описание/хар-ки
  в БД нет);
- рекомендации по **абсолютным стандартам Uzum Market** (`_recommendations_standalone`),
  БЕЗ выдуманных средних по конкурентам:
  • «📷 …не менее 5 качественных ракурсов»; • «📝 …от 300 символов (SEO)»;
  • «📋 >5 параметров»; • «⭐ …снижает конверсию в покупку».

Сравнительные рекомендации (`_recommendations_vs_competitors`, со средними по топу)
вызываются ТОЛЬКО когда конкуренты реально найдены — поэтому фраз «у конкурентов в
среднем N» / «у топа ~X» в режиме деградации не бывает. Бот не падает.

## Где взять данные селлера

В локальной БД богатых метрик нет (`user_products` хранит название/цену/остатки/
превью). Поэтому для честного сравнения карточка селлера скорится из её **публичной
карточки** (`fetch_product_card(product_id)`), как и конкуренты. Если публичная
недоступна — фолбэк на локальные метрики (фото=1, если есть `image_url`).
