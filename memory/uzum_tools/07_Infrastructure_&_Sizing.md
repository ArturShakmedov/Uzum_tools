 # 🏗️ Инфраструктура и планирование мощностей (Capacity Planning)

Назад к [[00_Index]]. Опирается на синк из [[05_API_Integration]], схему БД из
[[01_Database_Schema]] и двухфазную модель синхронизации (исправление C-1).

> Все цифры ниже — **эмпирические**, снятые с боевой БД для реального селлера
> `telegram_id=1245866204`: 587 SKU, история ~2 месяца (258 заказов, 209 накладных,
> 20 возвратов), а не теоретические оценки.

> 🟢 **СТАТУС СТЕКА (обновлено): миграция на highload-стек ВЫПОЛНЕНА.**
> - **БД → PostgreSQL** (драйвер `psycopg3`, синхронный), пул `pool_size=20 /
>   max_overflow=10 / pool_recycle=3600 / pool_pre_ping`. URL — `config.DATABASE_URL`.
> - **Двухфазный синк:** сеть параллельна (per-user lock + `FETCH_SEMAPHORE=10`, RAM),
>   а запись обёрнута в **короткий `DB_WRITE_SEMAPHORE=1`** (~1–3 с, сеть его НЕ держит).
>   На SQLite-деве спасает от «database is locked», на Postgres — дешёвая страховка
>   поверх MVCC (можно убрать ради макс. параллельности записи). Прежний глобальный
>   `SYNC_SEMAPHORE`, блокировавший и сеть, удалён.
> - **`raw_payload` → JSONB** (сжатие + GIN-индексы по содержимому).
> - **FSM → `RedisStorage.from_url(REDIS_URL)`** вместо `MemoryStorage` → бот
>   stateless, несколько инстансов за балансировщиком.
> - SQLite остался лишь дефолтом для локалки/CI. Тиры ниже описаны как план роста;
>   считать «БД = PostgreSQL» уже начиная с Tier 1 (SQLite-колонки — историческая
>   точка отсчёта, см. §5).

## 1. Аудит потребления ресурсов (замеры)

### 1.1 Сетевой I/O (httpx) — стоимость одного полного синка

Синк (`fetch_everything_from_uzum`) делает запросы по доменам. Замер карты вызовов
для селлера с ~600 SKU / 2 мес:

| Стадия | Метод | Запросов | Примечание |
|--------|-------|---------:|-----------|
| Заказы | `OrdersAPI.list_page` × 11 статусов | ~14–16 | size=50, по странице на статус + добор |
| Каталог | `ProductsAPI.iter_skus` | ~3–4 | size=100 **продуктов**/стр; 587 SKU ≈ 150–250 карточек |
| Остатки FBS (v3) | `StocksAPI.fbs_stock_map` | ~6 | `/v3/fbs/sku/stocks`, size=100, `ceil(587/100)` |
| Возвраты | `ReturnsAPI.iter_all` | ~1 | 20 < 50 → одна страница |
| Накладные (список) | `InvoicesAPI.iter_all` | ~11 | size=20, `ceil(209/20)` |
| Накладные (штрихкоды) | `InvoicesAPI.barcode_items` × N | **~209** | ⚠️ **1 запрос на КАЖДУЮ накладную** |
| **ИТОГО** | | **≈ 246** | доминанта — `barcode_items` (O(накладных)) |

- **Остатки FBS — через `/v3/fbs/sku/stocks`** (поле `amount`, конверт
  `payload.skuAmountList[]`). v2 отключён Uzum с 15.06.2026, бот полностью на v3.
  Стоимость: ~`ceil(SKU/100)` запросов (≈6 для 587 SKU) — под `FETCH_SEMAPHORE`.
  `fbo_stock` по-прежнему из каталога (`quantityActive`). См. [[05_API_Integration]].
- **Время синка:** при адаптивном троттлинге ~3 req/s минимум ~80 с «пола», с
  сетевой латентностью (~250–400 мс/запрос) реально **2–5 минут** на одного юзера.
- **Узкое место — `barcode_items`:** число запросов растёт линейно с числом
  накладных. У крупного селлера (1000+ накладных) синк = 1000+ запросов = 10+ мин.
  Кандидат на оптимизацию: батч-эндпоинт или кэш уже обработанных накладных.

### 1.2 Память на один синк (`SyncBundle` в RAM)

Фаза 1 собирает ВСЁ в `SyncBundle` (только dict'ы) до записи в БД:

| Источник | raw JSON | В Python-объектах (×3–5) |
|----------|---------:|-------------------------:|
| orders + orderItems | ~1.2 МБ | ~4–6 МБ |
| catalog (sku_barcodes) | ~1.5 МБ | ~5–7 МБ |
| barcodes | ~0.7 МБ | ~2–3 МБ |
| invoices + returns | ~0.5 МБ | ~2 МБ |
| **Итого на 1 синк** | **~4.5 МБ** | **~15–25 МБ RAM** |

> ✅ **Глобальный лимит параллельных синков внедрён.** `_user_locks` сериализуют
> только одного и того же юзера, а пиковую RAM ограничивает модульный
> **`FETCH_SEMAPHORE = asyncio.Semaphore(10)`** (`services/uzum_sync.py`), которым
> обёрнута сетевая фаза в `run_full_sync`. Пик RAM зафиксирован на **~200 МБ**
> (10 × ~20 МБ) даже при синке у 100+ юзеров одновременно; лишние ждут в очереди.
> Проверено: при 100 параллельных `run_full_sync` фактическая конкурентность = 10.
> Тюнинг под сервер: `пик_RAM ≈ FETCH_SEMAPHORE × 25 МБ` — на Tier 1 (RAM 2–4 ГБ)
> можно держать 10; на Tier 3 при широком канале поднимать до 16–24. Бонус: не
> хаммерит Uzum с одного IP (rate-limit обычно по IP/токену).

### 1.4 Event Loop (aiogram) — CPU на интерактив

| Действие | Стоимость | Где исполняется |
|----------|-----------|-----------------|
| Пагинация «Мои товары» | `GROUP BY sku_root` по индексу, <10 мс | `to_thread` |
| Карточка модели | агрегат по `list_products_by_root`, <10 мс | `to_thread` |
| Юнит-экономика | чистая арифметика, <1 мс | inline |
| ABC `build_model_analytics` | 2 SQL-агрегата (CTE) по ~300 строкам, <50 мс | `to_thread` |

**Вывод:** CPU на кнопки/расчёты **пренебрежимо мал** — все тяжёлые операции уже в
`asyncio.to_thread`, event loop не блокируется. Единственное реальное узкое место —
сетевой синк (см. §1.1). Фаза записи (`persist_to_db`) — ~4 транзакции, ~1000 строк,
**~1–3 с/юзер** и обёрнута в короткий `DB_WRITE_SEMAPHORE(1)` (сеть его не держит).
На PostgreSQL это лишь дешёвая страховка поверх MVCC: лок держится ~1–3 с записи,
а не минуты сетевого I/O, поэтому пропускная способность определяется ядрами/IOPS
Postgres, а не блокировкой. Прежний `SYNC_SEMAPHORE`, серииализовавший и сеть (потолок
к ~300–500 юзерам), снят. На SQLite-деве короткий лок убирает «database is locked».

## 2. Будущий функционал — маржинальная стоимость

| Фича | Диск | RAM | CPU/Сеть | Риск |
|------|------|-----|----------|------|
| **ИИ-генерации** (описания/контент через LLM-API) | ~0 (хранить только результат) | низкая | внешний HTTPS-вызов, латентность 1–5 с | стоимость токенов LLM, rate-limit провайдера; делать через `async`/`to_thread`, не блокировать loop |
| **ABC-анализ** | 0 (считается на лету) | низкая | <50 мс/запрос | уже реализован |
| **Рефералка** | ~0.1 КБ/строку (referrer→referred) | низкая | редкие select/insert | копеечная, отдельная таблица + индекс по `referrer_id` |

ИИ-генерации — единственная заметная новая нагрузка, но она **внешняя** (нагружает
LLM-провайдера, не наш сервер). На стороне бота — лишь исходящий вызов и удержание
ответа в RAM. Главное: вынести в воркер/`to_thread` и поставить свой троттлинг.

## 3. Три архитектурных тира

### 🟢 Tier 1 — Старт (10–50 активных селлеров)

Монолит: один процесс бота + SQLite в том же контейнере.

| Параметр       | Значение                                                                          |
| -------------- | --------------------------------------------------------------------------------- |
| **vCPU**       | 2 (shared)                                                                        |
| **RAM**        | 2–4 ГБ                                                                            |
| **NVMe**       | 40 ГБ                                                                             |
| **ОС**         | Ubuntu 22.04/24.04 LTS                                                            |
| **БД**         | SQLite (WAL) — с запасом                                                          |
| **Пример VPS** | Hetzner **CX22** (2/4 ГБ/40 ГБ) ≈ €4–5/мес · Selectel/Reg.ru: 2 vCPU/4 ГБ/40 NVMe |

- Диск: 50 × 30–50 МБ ≈ 1.5–2.5 ГБ данных — 40 ГБ хватает с большим запасом.
- RAM: база ~200 МБ + пик синков (5–10 параллельно × 20 МБ) ≈ 400 МБ. 4 ГБ — комфорт.
- **Лагов пагинации нет**: запросы по индексам <10 мс. SQLite single-writer не
  ощущается при <50 юзерах.

### 🟡 Tier 2 — Рост (100–500 активных селлеров)

Всё ещё монолит, но это **граница жизни SQLite**.

| Параметр | Значение |
|----------|----------|
| **vCPU** | 4 (желательно dedicated — стабильная латентность) |
| **RAM** | 8 ГБ |
| **NVMe** | 80–160 ГБ |
| **ОС** | Ubuntu 24.04 LTS |
| **БД** | SQLite до ~300 юзеров → **мост на PostgreSQL к 300–500** |
| **Пример VPS** | Hetzner **CPX31** (4/8 ГБ/160 ГБ) или **CCX13** (dedicated) · Selectel 4 vCPU/8 ГБ/120 NVMe |

**Хватит ли SQLite?** Технически — да, с нашими `DB_WRITE_SEMAPHORE`, индексами и
WAL до ~**300 активных**. Но:
- единственный писатель сериализует `persist_to_db` (~1–3 с/юзер); при «штормах»
  обновления кэша (много юзеров в одном 30-мин окне) очередь записи растёт;
- бэкап «на горячую» и масштабирование на несколько процессов невозможны.

**Вердикт:** на **верхней границе тира (400–500)** PostgreSQL становится практически
обязателен. Закладывайте миграционный мост заранее (см. §5). Требования к железу
почти не меняются (4 vCPU/8 ГБ), Postgres лишь снимает потолок параллелизма записи.

Обязательно к этому тиру: глобальный `FETCH_SEMAPHORE` (§1.2), ротация логов,
лимиты памяти контейнера (§4).

### 🔴 Tier 3 — Масштаб (1000+ селлеров, highload)

Монолит разбивается на роли. SQLite исключён — **только PostgreSQL**.

```
                 ┌──────────────┐   webhook/LB
   Telegram ───► │  Бот (aiogram)│◄────────────┐
                 │  stateless ×N │              │
                 └──────┬────────┘              │
                        │ enqueue sync          │ FSM/cache
                        ▼                        ▼
                 ┌──────────────┐         ┌────────────┐
                 │ Redis        │◄────────┤ воркеры     │
                 │ broker+FSM+  │  jobs   │ arq/Celery  │
                 │ cache        ├────────►│ ×N (синки)  │
                 └──────────────┘         └─────┬──────┘
                                                │ R/W
                                          ┌─────▼──────┐
                                          │ PostgreSQL │
                                          │ (managed)  │
                                          └────────────┘
```

| Роль | vCPU | RAM | Диск | Заметки |
|------|-----:|----:|------|---------|
| **Бот (aiogram)** | 2–4 | 4–8 ГБ | 20 ГБ | stateless, ×1–2 реплики через **webhook** (polling не масштабируется на реплики) |
| **PostgreSQL** | 4–8 | 16–32 ГБ | 200+ ГБ NVMe | managed (Neon/Selectel/Yandex) или dedicated; основной потолок производительности |
| **Воркеры синка** | 4–8 | 8–16 ГБ | 20 ГБ | **arq** (нативно async) или Celery; горизонтально масштабируются |
| **Redis** | 2 | 4 ГБ | 20 ГБ | FSM (`RedisStorage`), кэш отчётов/финансов, брокер задач, rate-limit |

- **Диск Postgres:** 1000 × 30–50 МБ ≈ 30–50 ГБ + индексы/WAL/раздутие → 200 ГБ с запасом.
- **FSM:** `MemoryStorage` → `RedisStorage` обязательно (иначе реплики бота не делят
  состояние диалогов).
- **Синки → воркеры:** тяжёлый `run_full_sync` уходит из процесса бота в очередь
  (arq), бот лишь ставит задачу и получает прогресс через Redis pub/sub.
- **Эконом-вариант старта Tier 3:** одна мощная dedicated-нода Hetzner **CCX33**
  (8 vCPU/32 ГБ) со всеми ролями в Docker Compose, затем вынос Postgres и воркеров
  на отдельные ноды по мере роста.

## 4. Технические рекомендации по деплою

- **Docker.** Один образ, разные команды: `python bot.py` (бот) и
  `arq worker.Settings` (воркер). `.env` — только через переменные окружения
  оркестратора (после C-2 `ENCRYPTION_KEY` обязателен, fail-fast без него).
- **Лимиты памяти контейнера.** Tier 1: `mem_limit: 1g`. Tier 2: `2–3g`.
  Воркеры Tier 3: `1.5g` на воркер (с учётом ~20 МБ/синк × конкурентность + база).
  Ставьте `FETCH_SEMAPHORE` так, чтобы `конкурентность × 25 МБ < mem_limit`.
- **Ротация логов.** Уже есть `RotatingFileHandler` (5 МБ × 7, см. `config.LOG`).
  В Docker дополнительно: `logging: { driver: json-file, options: { max-size: "10m", max-file: "5" } }`.
- **PostgreSQL (прод).** `DATABASE_URL=postgresql+psycopg://…`; пул уже в коде
  (`pool_size=20, max_overflow=10, pool_recycle=3600, pool_pre_ping`); `pgbouncer`
  при многих воркерах; PITR-бэкапы; отдельный persistent-volume под `PGDATA`.
- **Redis.** `REDIS_URL=redis://…` для `RedisStorage` (FSM); persistent или хотя бы
  AOF, чтобы не терять активные диалоги при рестарте.
- **SQLite — только дев/CI** (дефолт `DATABASE_URL`): файл `data/uzum.db` на
  persistent volume; в проде не используется.
- **Health-check.** `bot.py --dry-run` как readiness-проба (импорт + роутеры + БД,
  без polling).
- **Мониторинг:** RAM контейнера, длина очереди arq, p95 времени синка, частота 429
  от Uzum, размер/коннекты Postgres, заполнение пула (`pool_size`+overflow).

### Миграция на PostgreSQL — ✅ ВЫПОЛНЕНА

Стек уже переведён на Postgres (`psycopg3`) + Redis FSM. Что сделано в коде:

| Что | Как реализовано |
|-----|-----------------|
| Драйвер/URL | `postgresql+psycopg://…` из `config.DATABASE_URL` (SQLite — дефолт дев/CI) |
| Пул | `create_engine(pool_size=20, max_overflow=10, pool_recycle=3600, pool_pre_ping=True)` |
| Запись | короткий **`DB_WRITE_SEMAPHORE(1)`** только вокруг `persist_to_db` (~1–3 с); сеть параллельна. На Postgres — страховка поверх MVCC, на SQLite — против lock |
| `raw_payload` | `JSON().with_variant(JSONB(), "postgresql")` — JSONB на Postgres |
| FSM | `RedisStorage.from_url(config.REDIS_URL)` вместо `MemoryStorage` (бот stateless) |
| SQLite-специфика | PRAGMA-листенер (`journal_mode=WAL` и пр.) вырезан; `check_same_thread` остался только для SQLite-дева |
| Сырой SQL аналитики | `WITH lines … NULLIF`, `LOWER() LIKE … ESCAPE` — ANSI, проверено рендером под диалектом `postgresql` |

**Осталось для прода (вне кода приложения):**

1. **Alembic** — полноценные миграции схемы (аддитивные хелперы остаются только
   для SQLite-дева; на Postgres `create_all` строит таблицы+индексы из моделей).
2. **`pgbouncer`** перед Postgres при многих воркерах; **PITR-бэкапы**; отдельный
   инстанс БД.
3. Tier 3: вынести синки в воркеры **arq/Celery**, Redis — общий для FSM/кэша/брокера.

> Итог: потолок single-writer SQLite снят. До ~500 селлеров хватает монолита
> (бот + Postgres на одной ноде); 1000+ — разделение ролей (бот ╱ Postgres ╱
> воркеры ╱ Redis), см. Tier 3.
